# UI Reference Copier 产品理念

## 一句话说明

UI Reference Copier 是一个给 AI 编程工具使用的浏览器 UI 参考采集器。它不直接写代码，而是把浏览器真实渲染出来的元素信息整理成高质量提示词，帮助 Codex、Claude Code、Cursor 等工具更准确地还原页面。

它的目标不是替代设计师或前端工程师，而是解决一个非常具体的问题：当人把一个参考页面、原型页面或设计预览页面交给 AI 还原时，AI 经常只凭截图、HTML 或主观推断来写样式，结果很容易“看起来差不多，但细节总是不对”。

## 为什么需要这个工具

AI 写 UI 的能力越来越强，但 1:1 还原页面仍然困难。原因不是模型完全不懂 UI，而是它拿到的信息经常不够真实、不够精确，也不够贴近当前项目。

常见问题包括：

- 截图只能表达视觉结果，不能告诉模型真实的字体、间距、尺寸、DOM 层级和组件状态。
- HTML 只能描述结构，不能代表浏览器最终 computed style。
- 设计稿和实现页面之间经常隔着组件库、CSS token、响应式布局和运行时状态。
- 不同 UI 组件库的内部 DOM 差异很大，直接对比节点数量会误导模型。
- 后端工程师或不熟悉前端的人不一定会打开 DevTools，也不一定知道该复制哪些样式。
- 给模型的数据太少会还原不准，给太多 JSON 又会浪费 token，并且让模型抓不住重点。

UI Reference Copier 的出发点，就是把这些“需要前端经验才能整理的信息”变成一个普通人也能使用的浏览器插件工作流。

## 核心痛点

### 1. 只看截图还原不准

截图缺少真实渲染数据。模型可能猜出大概布局，但会在这些地方出错：

- 字号、字重、行高
- padding、margin、gap
- 容器宽高和相对位置
- 圆角、阴影、边框
- hover、active、展开态等组件状态
- 父级布局上下文

所以工具优先采集浏览器真实渲染结果，而不是只让模型凭视觉猜测。

### 2. 直接复制 DOM 又太脏

参考页面里经常有框架生成属性、埋点属性、组件库内部 class、运行时 style 和大量无关节点。如果直接把完整 HTML 或 JSON 塞给模型，模型容易照搬错误的 class，甚至创建当前项目根本不存在的样式。

因此工具默认输出 token 友好的提示词，只保留对还原有帮助的信息；完整 JSON 和完整 computed style 作为高级选项单独提供。

### 3. 不同 UI 组件库不能硬比 DOM

例如参考页使用 Naive UI，当前项目使用 Element Plus。两个菜单看起来都是侧边栏菜单，但内部 DOM 可能完全不同：

- 参考页可能是大量 `div`
- 当前页可能是 `ul / li / span`
- 参考页可能有 `--n-*` 变量
- 当前页可能使用 Element Plus class 和 CSS 变量

如果工具只告诉模型“div 少了多少、li 多了多少”，模型就会被带偏，开始重写 DOM 或复制参考组件库变量。

所以工具把菜单、导航这类组件当成“语义结构”处理：重点是菜单项、层级、顺序、图标、选中态、展开态，而不是内部 DOM tag 是否一致。

### 4. 普通用户需要一个浏览器插件，而不是复杂流程

这个工具很重视“后端也能用”。理想流程应该是：

1. 打开参考页面。
2. 点插件。
3. 选中元素。
4. 复制提示词。
5. 粘贴给 Codex / Claude Code。

用户不需要理解 computed style、DOM path、CSS 变量、组件库内部结构，也不需要在 DevTools 里手工找属性。

## 设计原则

### 1. 以当前项目为修改目标

工具始终区分“参考页面”和“当前实现页面”。

参考页面的信息只用于理解视觉目标，不应该被直接照搬。当前页面的 selector、组件结构和项目样式体系才是模型应该修改的目标。

因此对比提示词会保留当前目标 selector，同时提醒模型不要复制参考页 class、id、selector 或组件库专属 DOM。

### 2. 真实渲染优先

工具采集的是浏览器运行后的信息，包括：

- 元素 rect
- computed style
- 父级布局上下文
- 关键子元素样式
- CSS 变量
- 图标细节
- 结构相似度

这比纯 HTML 或截图更接近模型真正需要的实现依据。

### 3. 默认精简，高级可展开

默认提示词应该让模型抓住重点，而不是把所有数据倒进去。

所以工具采用分层输出：

- 默认复制：给关键视觉信息和修复建议。
- 详细复制：用于复杂问题排查。
- JSON / 完整 computed CSS：只在需要时手动复制。
- 子元素采样：可选不采集、精简、标准、详细。
- 结构采样：可选精简、中等、全量。
- 样式对比范围：可选只对比根元素或包含子元素。

这背后的原则是：少给无关信息，多给决策线索。

### 4. 先结构，后样式

很多 UI 还原失败，不是颜色或字体的问题，而是结构层级不一致：

- 参考选的是外层卡片，当前选的是内部文字。
- 参考有 header / body / footer，当前被拍平成几个 div。
- 参考有进度条、图表、操作区，当前缺少对应结构。
- 菜单组件的层级、展开态、选中态不一致。

如果结构不一致，继续修样式很容易越改越偏。

因此工具提供独立的结构对比，用来先判断两个元素是不是同一层级，再决定是否进入样式修复。

### 5. 对跨组件库场景保持克制

工具不应该要求模型把当前项目改成参考页面的组件库。更合理的方式是：

- 保留当前项目已有组件。
- 保留当前项目路由配置、菜单数据源和 design token。
- 把参考页面作为视觉目标。
- 把参考组件库变量翻译成当前项目可用的样式或 token。

这也是菜单组件对齐模式存在的原因。

### 6. 让提示词像任务说明，而不是数据转储

最终给 AI 的内容应该像一个清晰的工程任务：

- 当前要修改哪个元素
- 参考视觉目标是什么
- 当前和参考的关键差异是什么
- 哪些信息不能照搬
- 应该优先改结构、样式、颜色还是菜单数据
- 如果不确定，应该如何验证

工具的价值不只是采集数据，更是把数据组织成 AI 能执行的任务上下文。

## 核心能力

### 单元素采集

适合从参考页复制某个按钮、卡片、容器或局部 UI 的真实渲染信息。

用户可以选择是否采集子元素，减少 token 或保留内部细节。

### 跨页面对比

适合参考页和当前实现页之间做一对一差异分析。

它会输出：

- 整体边界差异
- 字体、颜色、背景、边框、圆角、阴影
- 父级布局上下文
- 可选子元素差异
- 图标差异
- 结构风险提示

### 多组元素对比

适合一个页面里有多个区域需要分别对齐，例如多个指标卡片、多个表单区、图表和表格区域。

多组对比的价值是让模型知道“每一组对应哪个目标”，而不是把整个页面混在一起改。

### 结构对比

结构对比负责回答一个关键问题：现在是不是选中了同一类东西？

它不优先修颜色和字体，而是检查：

- 根节点
- display
- 子元素数量
- tag / role 分布
- 文本节点
- 媒体节点
- 父级布局
- 菜单/导航语义

### 菜单组件对齐

菜单是特殊场景，因为不同 UI 库内部 DOM 差异很大。

工具会尽量把菜单理解成语义树：

- 菜单项文本
- 层级
- 顺序
- 展开态
- 选中态
- 图标位置
- 箭头位置

这能避免模型被 `div / ul / li / span` 数量差异带偏。

### 取色

取色是独立工作流，不要求用户选择整个元素。

它更像吸管：

- 从屏幕像素读取颜色。
- 再选择目标元素。
- 指定应用到背景、文字、边框、图标、阴影或主题变量。

这个功能适合只同步颜色，而不修改布局和结构。

## 典型使用方式

### 还原一个参考组件

1. 在参考页面选中元素。
2. 必要时选择父级或回退。
3. 复制给 AI。
4. 让 AI 在当前项目中找到对应组件和样式文件。

### 修一个已经实现但不够像的页面

1. 在参考页设为参考。
2. 到当前实现页选中对应元素。
3. 对比参考。
4. 如果结构风险高，先做结构对比。
5. 结构一致后，再处理样式差异。

### 对齐不同 UI 库的菜单

1. 切到结构对比。
2. 结构采样深度选全量。
3. 保存参考菜单和当前菜单。
4. 看菜单语义差异，而不是 DOM tag 差异。
5. 在样式对比中先选择只对比当前元素。
6. 根容器对齐后，再包含子元素排查细节。

### 只同步一个颜色

1. 切到取色。
2. 用吸管吸取颜色。
3. 到目标页面选择目标元素。
4. 选择颜色应用位置。
5. 复制颜色提示词给 AI。

## 这个工具不是什么

UI Reference Copier 不是自动代码生成器。它不会直接改项目源码。

它也不是完整视觉 diff 工具。它不会保证像素级一致，也不会替代人工验收。

它更像一个“AI UI 修复上下文生成器”：把浏览器、页面、组件、样式和结构里的关键信息整理好，让 AI 少猜一点，让人少解释一点。

## 适合的人

- 后端工程师：不会细调 CSS，也可以把真实 UI 信息交给 AI。
- 前端工程师：减少手工查 computed style 和对比样式的时间。
- 独立开发者：快速从参考站点或模板页面借鉴 UI。
- 使用 AI 编程工具的人：给模型更可靠的上下文，减少反复试错。
- 设计还原场景：把原型页面、设计预览页面和当前实现之间的差异说清楚。

## 后续方向

工具后续可以继续增强几个方向：

- 元素截图裁剪，把视觉证据和渲染数据一起交给模型。
- 菜单和表格等复杂组件的更精确语义识别。
- 自动展开可展开菜单后再采集，并在采集后恢复状态。
- 更智能的子元素匹配，减少不同 UI 库之间的错配。
- 与本地项目源码建立更强关联，例如提示模型优先查找菜单数据源、路由配置或 design token。
- 提供二次验证工作流，让 AI 修改后重新采集并继续对比。

## 最终目标

这个工具最终想解决的不是“复制 CSS”这么小的问题，而是让 AI 编程工具更可靠地参与 UI 还原工作。

它希望把原本只有熟练前端才能说清楚的东西，变成普通用户也能一键提供的上下文：

- 参考是什么
- 当前是什么
- 差异在哪里
- 哪些东西不能照搬
- 应该优先改哪里

当这些信息足够清楚，AI 才更可能写出接近真实目标的 UI，而不是写出一个“看起来像，但哪里都差一点”的页面。
